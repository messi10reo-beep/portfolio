How it works

実装解説Under the hood, in plain words

このデモがどう動いているかを、コードが分からない方にも分かるように解説します。全体構成・データの保存・FAQ検索・分類ロジック・APIなしで動く理由・今後の拡張方針の順です。

01

全体構成

「集める → AIで整理する → 人が確認して返す」の3ステップ。

入力予約 / 問い合わせ / FAQ質問
→
AI整理分類・検索・たたき台生成
→
集約ダッシュボードに保存
→
人が確認最終判断・返信・確定

画面は、公開サイト（LP・予約・FAQ・問い合わせ）と、スタッフが見る管理ダッシュボードに分かれています。AIは中間で「下ごしらえ」をするだけで、最後は必ず人が確認する設計です。

02

データ保存の方法

このデモではブラウザ内の localStorage を使用。

予約や問い合わせは、サーバーではなくあなたのブラウザの中（localStorage）に保存されます。だからログイン不要・APIキー不要で、送信内容がそのままダッシュボードに反映されます。外部に送信されないため、デモとして安全に触っていただけます。

データは下記のような「型」で整理しています（TypeScript設計に準拠）。

型主なフィールド
Bookingid / name / email / phone / menu / firstChoice / secondChoice / symptoms / isFirstVisit / note / category / aiSummary / createdAt / status
Inquiryid / name / email / type / message / category / aiSuggestedReply / createdAt / status
FAQItemid / category / question / answer / keywords

※ category / aiSummary / aiSuggestedReply がAIによって自動付与される部分です。本番では Supabase などの永続DBに置き換えるだけで、画面ロジックはほぼそのまま使えます。

03

FAQ検索の仕組み（簡易RAG風）

あらかじめ用意したFAQ15項目から、関連度の高いものを探して答えます。

1
質問を正規化

大文字・全角・空白・記号を整え、表記ゆれを吸収します。

2
キーワード照合でスコア化

各FAQの「キーワード」が質問に含まれるかを見て、点数をつけます（長い語ほど高配点）。

3
最上位を回答＋根拠提示

一番点数の高いFAQで回答し、関連FAQを「根拠」として表示します。

4
自信がなければ案内

点数が低いときは断定せず、「お問い合わせください」と誘導します。

// 質問に含まれるキーワードでスコアリング（簡易RAG風） faq.keywords.forEach(function (kw) { if (query.indexOf(kw) !== -1) score += Math.min(kw.length, 4); }); // 最上位スコアが低ければ断定せず問い合わせへ誘導
04

問い合わせ・予約の分類ロジック

キーワードのルールベースで、カテゴリと確認ポイントを決めます。

予約は、症状文に「腰」が含まれれば腰痛相談、「姿勢・猫背」なら姿勢改善、「肩・首」なら肩こり・首こり相談… のように分類し、それぞれカウンセリングで確認すべき項目を提示します。

問い合わせは、選択された種別（予約 / 料金 / 症状 / キャンセル / その他）をもとにカテゴリを決め、種別が「その他」のときは本文から推定します。カテゴリごとに返信文のたたき台を生成します。

// 予約：症状文からカテゴリ判定（優先順にマッチ） if (/腰|ぎっくり|坐骨/.test(text)) cat = "腰痛相談"; else if (/姿勢|猫背|反り腰/.test(text)) cat = "姿勢改善"; else if (/肩|首|こり|張り/.test(text)) cat = "肩こり・首こり相談"; else cat = "一般相談";

※ ルールベースは「なぜその分類になったか」が説明できるのが強みです。誤りを見つけたら、その場でルールを直せます（ブラックボックスになりにくい）。

05

APIキーなしでも動く理由

Q. なぜAPIキーが要らないの？

FAQの知識・分類ルール・返信テンプレートを、すべてアプリ内に同梱しているからです。外部のAIサービスを呼び出さないので、キーも通信も不要で、その場ですぐ動きます。

Q. 本物のAIじゃないの？

このデモは「AIらしい体験」をルールベースで再現したものです。仕組み（集める→整理→人が確認）はそのままに、整理部分を本物のLLM APIへ差し替えれば、回答の精度を上げられます。

Q. 安全面は？

入力はブラウザ内にのみ保存し外部送信しません。AIの回答範囲をFAQに限定し、医療的な断定は避ける設計です。最終確認は必ず人が行います。

06

今後API連携する場合の拡張方針

  • →FAQ回答：キーワード検索を、ベクトル検索＋LLM（Claude / OpenAI）に置き換え、表現ゆれや複合質問に強くする。
  • →データ保存：localStorage を Supabase などの永続DBに変更。複数スタッフで共有・検索・CSV出力。
  • →予約連携：Google Calendar / LINE と連携し、空き状況の自動反映・確定通知を実現。
  • →運用：AIの提案精度をログで計測し、確認・修正された内容を学習データとして改善に回す。

画面とデータの「型」を先に固めてあるため、これらの拡張は裏側の差し替えで進められます。
